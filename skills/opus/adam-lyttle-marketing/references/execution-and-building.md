# Execution And Building Reference

Source: `sources/Execution & Building (Part 2 of 3).srt`

This reference distills Adam Lyttle's process for turning a validated idea into a shipped app without getting trapped in endless building.

## Operating Principle

Solo builders need constraints because there is no boss, team, or external deadline. Without constraints, projects expand, refactors multiply, polish consumes the schedule, and the app never reaches users.

The objective is to ship a deliberate first version, not to perfect every surface before launch.

## Deadline And Constraints

Use a self-imposed deadline with consequences. The more real and specific the deadline, the better.

Good deadlines:

- A specific launch date.
- A seasonal demand window.
- A content schedule or public launch moment.
- A date tied to a real opportunity cost.

The deadline clarifies how much time each feature deserves. If launch is days away, cosmetic inconsistency may move to the then list instead of blocking release.

## Launch-Critical Work And Then List

Separate work into:

- Launch-critical: core feature, main flow, monetization path if required, first-run usability, score/reward loop, app store readiness.
- Then list: cosmetic polish, nice-to-have features, minor visual inconsistencies, extra unlocks, secondary menus, non-critical improvements.

Adam's rule of thumb: get the app to the useful 80% first, then work through the final 20% only after the launch-critical version exists.

## Code Massaging

Build rigid functionality first:

- No bells.
- No cosmetics.
- No whistles.
- Just the feature working in its simplest form.

Then refine each area one by one. This makes the app behave like an advanced prototype: every feature can be tested in simple form before large time investment.

Why this matters:

- New ideas appear during the build.
- Some features prove unnecessary.
- Some interactions feel wrong only after use.
- Less time is wasted when a feature must be replaced or removed.

## Focus Rule

Focus on one project at a time during the build phase. If new ideas appear, write them down and return later. The active objective is to release the current app.

## Feedback

Get feedback early from family, friends, children, users, or other developers. Watch how people use the app when handed it without explanation.

Feedback principles:

- Test accessibility and engagement.
- Observe actual behavior, not just comments.
- Do not make testers feel uncomfortable by defending unfinished work.
- Treat feedback as data even when the app is still rough.

Swipe the Cat example:

- The initial game start required timing with a beat.
- A younger tester struggled and could not score.
- Adam removed the beat from the demo/start flow so users could begin without timing pressure.

## Flow And Core Loop

Well-coded does not mean enjoyable. The app must flow.

Adam's flow pattern:

- Put users straight into the thing they came to do.
- Avoid unnecessary menus.
- Avoid hiding the main use case on secondary screens.
- Reduce friction.
- Throw users back into the core loop quickly.
- Pause only where the pause serves a purpose.

Swipe the Cat flow:

- Title screen, then immediately the game.
- Game over, then score tally.
- Score tally is the intentional pause and screenshot/share moment.
- Continue leads to gift/unlock progress.
- Then users return directly to the main game.

## Design Vision

AI-generated or AI-assisted apps can become visually bland if the builder has no design vision. The builder is responsible for the final feel.

Build a clear visual direction from:

- The original inspiration.
- A visual board or whiteboard.
- References from other apps or media.
- The emotional reaction the app should create.
- The content users might share.

Swipe the Cat visual direction:

- Inspired by hand-drawn animation in the thumb-swipe trend.
- Goal: every element feels hand-drawn and rendered on a sketch pad.
- Reason: in a feed, a screen recording should look authentic and not like a generic app.

## Common Mistakes

- Endless refactoring before launch.
- Waiting for the app to feel perfect.
- Spending launch time on cosmetic details.
- Building all features deeply before testing basic behavior.
- Letting new ideas distract from the active project.
- Defending rough work instead of listening to testers.
- Adding menus or explanations that separate users from the core action.
- Letting AI decide the product's visual identity.

## Planning Checklist

1. Define the real deadline and consequence.
2. List launch-critical tasks.
3. Move polish and nice-to-haves to the then list.
4. Build the rigid version of each feature first.
5. Test each core area in simple form.
6. Gather early feedback by observing real use.
7. Remove friction from the main flow.
8. Identify intentional pauses and why they exist.
9. Define the visual direction before final polish.
10. Ship the first deliberate version.
